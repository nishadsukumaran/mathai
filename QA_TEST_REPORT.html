AUTH-01 Client API always sends hardcoded "Bearer dev-stub" — authenticated calls never work
📁 apps/web/lib/clientApi.ts
🔗 apps/web/lib/auth.ts (authOptions)
Impact: Every authenticated call from the browser — fetching the student profile, saving profile changes, sending Ask MathAI queries — goes out with the token "Bearer dev-stub". The Express API validates this token, and in production (where no dev-stub bypass exists) every API call returns 401. The entire authenticated experience is broken.

clientApi.ts reads the access token via session?.["accessToken"]. But authOptions.callbacks.session in auth.ts only copies token.id onto the session object. The field accessToken is never written, so it is always undefined. The fallback returns the literal string "Bearer dev-stub" for every request.

// apps/web/lib/clientApi.ts — getToken() 12const token = session?.["accessToken"]; // ❌ never set 13return token ? `Bearer ${token}` : "Bearer dev-stub"; // always falls back // apps/web/lib/auth.ts — callbacks.session 41callbacks: { 42 async session({ session, token }) { 43 (session.user as { id?: string }).id = token.id as string; 44 // ❌ session.accessToken is never set here 45 return session; 46 }, 47}
Fix — two changes required: 1. In authOptions.callbacks.jwt, store the JWT string (or a signed token) on token.accessToken.
2. In authOptions.callbacks.session, propagate it: (session as any).accessToken = token.accessToken.
Or simplify by calling the Next.js internal /api/auth/token route already present in the codebase (apps/web/app/api/auth/token/route.ts) to get a short-lived token and cache it in clientApi.ts.
AUTH-02 Confidence level scale mismatch — slider broken on load, sends value 20× too high on save
📁 apps/web/app/profile/ProfilePageContent.tsx
🔗 api/services/profileService.ts
Impact: The confidence slider renders at maximum (5) for any student with moderate confidence (e.g., 60/100 stored as 60, shown as 5 capped). On save, if student moves slider to 3, it POSTs 3. The backend multiplies by 20 and stores 60. But on next load it renders as 5 again. The field is permanently uneditable in any meaningful way.

The database stores confidenceLevel as a 0–100 EWMA float (e.g., 62.5). The Profile UI renders a slider with min=1 max=5. The useEffect that populates state sets it directly from the raw DB value — no division by 20. Any value over 5 is rendered at 5 (slider max), so the UI shows full confidence for every student above the beginner tier.

// ProfilePageContent.tsx — useEffect populating slider state 58useEffect(() => { 59 if (!profile) return; 60 setConfidence(profile.confidenceLevel ?? 3); // ❌ sets 62 on a 1-5 slider → renders as 5 61}, [profile]); // Correct approach setConfidence(Math.round((profile.confidenceLevel ?? 60) / 20)); // maps 0–100 → 1–5
Fix: In the useEffect in ProfilePageContent.tsx, divide the raw DB value by 20 and clamp to [1, 5]: setConfidence(Math.max(1, Math.min(5, Math.round((profile.confidenceLevel ?? 60) / 20)))). The save path already sends the 1–5 value which profileService multiplies back to 0–100, so only the load side needs fixing.
GRADE-01 All practice sessions, hints, and explanations are hardcoded to Grade 4 — grade profile ignored
📁 api/controllers/practiceController.ts
🔗 api/middleware/auth.ts (requireStudent)
🔗 apps/web/lib/auth.ts (authOptions.callbacks.jwt)
Impact: A Grade 2 student gets Grade 4 multiplication questions. A Grade 8 student gets Grade 4 fractions. The entire personalisation promise of MathAI is broken at the practice layer — the single most important feature of the product.

practiceController.ts reads the grade from req.student!.grade. req.student is populated by the requireStudent middleware which decodes the JWT. But the JWT is built in authOptions.callbacks.jwt, which only writes token.id (from the NextAuth token). The user's gradeLevel from the DB is never placed into the JWT, so req.student.grade is always undefined, and the fallback ?? "G4" fires every time.

// api/controllers/practiceController.ts 24const grade = (req.student!.grade ?? "G4") as Grade; // ❌ always G4 — grade never in JWT // apps/web/lib/auth.ts — jwt callback 31async jwt({ token, user }) { 32 if (user) token.id = user.id; 33 // ❌ gradeLevel from DB never added to token 34 return token; 35}, // Fix: fetch grade in jwt callback and embed it async jwt({ token, user }) { if (user) { token.id = user.id; const dbUser = await prisma.user.findUnique({ where: { id: user.id }, select: { gradeLevel: true } }); token.gradeLevel = dbUser?.gradeLevel ?? "G4"; } return token; },
Fix — two steps:
1. In authOptions.callbacks.jwt, fetch gradeLevel from Prisma on user sign-in and store as token.gradeLevel.
2. In authOptions.callbacks.session (or equivalent token middleware), propagate it to the signed JWT used by the Express API.
3. Also update token when grade is changed via PATCH /profile — either force a JWT re-issue or look up grade from DB fresh in requireStudent middleware.

NAV-01 Profile page is unreachable on mobile — not included in bottom navigation bar
📁 apps/web/components/mathai/AppNav.tsx
Impact: Any student on a mobile device (the primary device for this age group) cannot navigate to their profile to change grade, learning pace, or preferences. Combined with GRADE-01 above, this means mobile users have no path to fix incorrect personalisation.

AppNav.tsx defines a NAV_ITEMS array with 5 entries: Dashboard, Practice, Ask MathAI, Progress, and Leaderboard. On desktop, a sidebar also renders a "Profile" link. On mobile, only the bottom bar with NAV_ITEMS is shown. Profile is absent from that array.

Fix: Add a Profile entry to NAV_ITEMS (or conditionally render it in the mobile bar separately). Consider using a 👤 icon. If 6 items are too many for mobile, replace one lower-priority item (e.g., Leaderboard) or use a compact "More" overflow approach.
SIGNUP-01 No grade captured at signup — every new user defaults to Grade 4 forever
📁 apps/web/app/api/auth/signup/route.ts
🔗 apps/web/app/signup/page.tsx (signup UI form)
Impact: Every student who signs up has no grade set. Combined with GRADE-01 (JWT never carries grade), this means a brand new user's first practice session is Grade 4. There's no prompt to select grade, and given NAV-01 on mobile, they may never find the profile page to fix it.

The signup API route creates a User with name, email, and hashedPassword. No gradeLevel field is set. The signup form UI has fields for name, email, and password — no grade selector. The StudentProfile is created in a follow-up Prisma call with all defaults.

Fix: Add a grade selector step to the signup form (a simple 1–10 picker or dropdown works well for this age group). Pass the selected grade to the API route and set gradeLevel on User creation. Alternatively, add a mandatory onboarding screen shown after first login that captures grade before proceeding.
DB-01 Multiple PrismaClient instances — connection pool leak risk under load
📁 apps/web/lib/auth.ts
📁 apps/web/app/api/auth/signup/route.ts
🔗 apps/web/lib/prisma.ts (singleton — exists but unused here)
Impact: In development with Next.js hot-reload, each module re-evaluation creates a new PrismaClient. Each client opens a database connection pool. In production, this is less frequent but still wasteful. Prisma warns explicitly about this pattern in its Next.js docs and recommends a singleton.
// apps/web/lib/auth.ts 3const prisma = new PrismaClient(); // ❌ new instance per module load // apps/web/app/api/auth/signup/route.ts 3const prisma = new PrismaClient(); // ❌ another new instance // The singleton already exists — just import it import { prisma } from "@/lib/prisma"; // ✓ use shared singleton
Fix: Replace both new PrismaClient() instantiations with import { prisma } from "@/lib/prisma". The singleton is already defined at apps/web/lib/prisma.ts with the recommended Next.js global caching pattern.
PRACTICE-01 Practice session start request does not include grade — controller always receives null
📁 apps/web/components/mathai/PracticeContainer.tsx
🔗 api/controllers/practiceController.ts

Even if GRADE-01 is fixed so the JWT carries gradeLevel, the POST body sent by PracticeContainer to /api/practice/start includes topicId and mode but no grade field. The controller currently reads grade from req.student.grade (the JWT-decoded value), so this is a secondary path — but including it in the body would make the flow explicit, debuggable, and independent of JWT state. Also noted: an earlier version of the container had a hardcoded grade: "G4" in the body which was an additional override risk.

Fix: In PracticeContainer.tsx, read grade from the session or from the loaded student profile and include it in the POST body. Update the Zod schema in practice.routes.ts to accept an optional grade field. This makes the grade path explicit and easier to test.
P2 — Medium

Functional issues and incomplete features. Fix before public launch.

AUTH-03 Practice page has no server-side auth guard — unauthenticated users see a flash before redirect
📁 apps/web/app/practice/page.tsx
🔗 apps/web/middleware.ts

apps/web/app/practice/page.tsx is a client component that uses useSession() with a redirect if unauthenticated. This causes a flash of the practice UI before the redirect fires. Other pages (e.g., Dashboard, Leaderboard) use getServerSession() in a server component and redirect on the server. The practice page is inconsistent and provides a worse user experience.

Fix: Either (a) convert practice/page.tsx to a server component using getServerSession() and redirect(), or (b) ensure middleware.ts covers the /practice route (check the matcher config).
DATA-01 Hardcoded "user-alice-001" fallback in Dashboard and Progress — real users see no data
📁 apps/web/components/mathai/DashboardContainer.tsx
📁 apps/web/components/mathai/ProgressContainer.tsx

Both containers extract userId from session.user with a fallback to the literal string "user-alice-001". If the session user has no id (which happens when AUTH-01 is in play and sessions aren't returning the full user object), every real user's dashboard loads data for a fictional seed user. Their XP, streaks, and progress data appears as Alice's data.

Fix: Remove the fallback string entirely. If userId is undefined, redirect to login or show an error state. Do not silently serve another user's data.
PROFILE-01 Profile page has no name editing UI — name change is impossible despite API supporting it
📁 apps/web/app/profile/ProfilePageContent.tsx
🔗 api/routes/profile.routes.ts (PATCH accepts name)

The profile page renders the user's display name as static text. The PATCH endpoint at /api/profile accepts a name field in the request body, and profileService.updateProfile() persists it to User.name. But the UI has no editable input field for it. A student who misspelled their name at signup has no way to fix it.

Fix: Add a text input for display name to the profile form, wired into the existing save flow alongside grade, learning pace, and explanation style.
PROFILE-02 Profile save errors are silently discarded — user receives no feedback on failure
📁 apps/web/hooks/use-profile.ts
📁 apps/web/app/profile/ProfilePageContent.tsx

use-profile.ts has an error state field in its return object. On a failed save, this state is updated. But ProfilePageContent.tsx never reads the error field — there is no error message, toast, or visual indicator of any kind. If the save fails (network error, 401, 500), the user sees the "Saved!" button animation and nothing else. They have no idea their changes were not persisted.

Fix: In ProfilePageContent.tsx, destructure error from useProfile() and render a visible error message or toast when it is truthy. An inline red alert below the Save button is the simplest approach.
P3 — Low / UX Polish

Minor UX gaps and missing polish. Fix before public launch if time allows.

UX-01 No custom 404 page — Next.js default shown on bad URLs
📁 apps/web/app/not-found.tsx (missing)

Navigating to an invalid route (e.g., /mathai/foo) shows the default Next.js 404 page. For a product aimed at children, a branded, friendly 404 with a "Go back home" button is important for user confidence.

Fix: Create apps/web/app/not-found.tsx with a fun math-themed message and a prominent link back to /dashboard.
UX-02 Ask MathAI textarea does not auto-resize — long questions feel cramped
📁 apps/web/app/ask/AskPageContent.tsx

The question input is a fixed-height <textarea>. As the student types a longer question, scrollbars appear inside the box rather than the box growing to fit. This feels clunky, especially on mobile.

Fix: Add an onInput handler that sets element.style.height = "auto" then element.style.height = element.scrollHeight + "px". Or use the react-textarea-autosize library (already in many Next.js starters).
UX-03 Page does not scroll to top on route change — content appears mid-page
📁 apps/web/app/layout.tsx

When navigating between pages (e.g., Practice → Dashboard), Next.js App Router preserves scroll position in some scenarios. If the user scrolled down on Practice, Dashboard opens mid-scroll. There is no scroll-reset mechanism in the root layout or route transitions.

Fix: Add a minimal client component to the root layout that calls window.scrollTo(0, 0) on usePathname() change. This is a common Next.js App Router pattern.
DATA-02 Leaderboard is fully mocked static data — does not reflect real students
📁 apps/web/app/leaderboard/page.tsx

The leaderboard always renders Alice, Marcus, Priya, Leo, and Zoe with hardcoded XP values. No real student data is shown. The page does include a footer note saying real-time is "coming soon" — so this is a known placeholder, but worth tracking as a formal backlog item.

Fix: Implement a GET /gamification/leaderboard?scope=class&limit=10 endpoint that queries the top students by StudentProfile.xp and returns anonymised or real names based on privacy settings.
