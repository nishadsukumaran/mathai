Critical Bugs

3 issues
DATA-02 — Leaderboard page still carries the `user-alice-001` fallback
DATA-01 from Wave 7 fixed the dashboard and progress pages but missed leaderboard/page.tsx. Line 52 reads: const userId = … ?? "user-alice-001". This means an authenticated user with a real session whose session.user.id is somehow undefined will silently call /gamification/dashboard under user-alice-001, showing another user's XP and streak data on their leaderboard header — a real data leak in production.
apps/web/app/leaderboard/page.tsx : line 52 
// Current (broken)
const userId = (session.user as { id?: string }).id ?? "user-alice-001";

// Fix — mirror the dashboard/progress pattern
const userId = (session.user as { id?: string }).id;
if (!userId) redirect("/auth/signin");
P0 — Data Leak
PROFILE-03 — Dashboard ProfileModal saves wrong confidence value + doesn't refresh JWT grade
Two bugs in ProfileModal used from DashboardView:

(a) Confidence value mismatch: The modal initialises state with data.confidenceLevel which is the raw 0–100 EWMA value from the DB. The slider is min=1 max=5, so HTML clamps the display to 5 regardless. If the user doesn't touch the slider and saves, confidenceLevel: 60 (or whatever raw value) is sent to the API — instead of the correct 1–5 value. The fix applied in Wave 7 (Math.round(raw / 20)) only made it into ProfilePageContent, not ProfileModal.

(b) Grade change doesn't update JWT: ProfileModal.onSave calls saveProfile(patch) but never calls updateSession({ grade }). So after a grade change via the dashboard modal, the user's JWT still carries the old grade and every subsequent practice/hint API call uses stale grade data.
apps/web/components/mathai/profile-modal.tsx : lines 83, 188 
// (a) Confidence — add the same mapping as ProfilePageContent
const rawConf = data.confidenceLevel ?? 60;
const [confidence, setConfidence] = useState<number>(
  Math.max(1, Math.min(5, Math.round(rawConf / 20)))
);

// (b) grade sync — after onSave call updateSession
const { update: updateSession } = useSession();
async function handleSave() {
  …
  await onSave({ learningPace: pace, style, confidenceLevel: confidence });
  await updateSession({ grade: data.grade });  // sync JWT
  onClose();
}

API Handling Issues

4 issues
API-01 — clientApi.getToken() silently falls back to "dev-stub" on 401
When /api/auth/token returns 401 (user's session has expired mid-session), clientApi.ts catches the non-ok response and returns "Bearer dev-stub". All subsequent API calls fire with the stub token. The Express API rejects them, callers get null, and pages silently break. There is no UX prompt to re-authenticate.
apps/web/lib/clientApi.ts : getToken() 
// After a 401, instead of dev-stub, redirect to sign-in
async function getToken(): Promise<string> {
  try {
    const res = await fetch("/api/auth/token");
    if (res.ok) { … return `Bearer ${token}`; }
    if (res.status === 401) {
      window.location.href = "/auth/signin";
      return "";   // unreachable
    }
  } catch { /* network */ }
  return "Bearer dev-stub";
}
P1 — Auth Regression
API-02 — PracticeContainer makes raw fetch calls with no timeout on submit/hint
startSession correctly uses an AbortController with a 30s timeout and gives a friendly "server warming up" message. But submitAnswer and getHint have no timeout at all — a slow AI response can leave the student's answer stuck in "Checking…" indefinitely.
apps/web/containers/PracticeContainer.tsx : submitAnswer, getHint
P2 — Reliability
API-03 — useProfile has no refetch/retry mechanism
useProfile fetches on mount with a cancelled-flag pattern, but exposes no refetch callback. If the initial load fails (transient network hiccup), the user must do a full page reload to recover. Compare with React Query hooks (used in useDashboard, useProgress) which have automatic retry and expose refetch for free.
apps/web/hooks/use-profile.ts
P2 — Reliability
API-04 — AskPageContent does not check res.ok before parsing JSON
clientPost<AskMathAIResponse>("/tutor/ask", …) returns null on non-ok responses (handled correctly in clientApi). However, the component sets error only when data is falsy but doesn't distinguish between a network error and a meaningful API error message. The student always sees "MathAI couldn't answer right now" with no additional context.
apps/web/app/ask/AskPageContent.tsx : handleSubmit

Component Inconsistencies

5 issues
CONSIST-01 — ProfileModal and ProfilePageContent are duplicated settings UIs
Both components implement learning pace selector, explanation style pills, and confidence slider independently. They are now diverging: ProfilePageContent has the name editor and grade picker; ProfileModal does not. ProfilePageContent has the correct confidence mapping; ProfileModal does not (see PROFILE-03). Any future settings field added to one needs to be added to both manually. These should share a <ProfileSettings> sub-component.
profile-modal.tsx vs app/profile/ProfilePageContent.tsx
P2 — Architecture
CONSIST-02 — Two API clients with different auth and error handling strategies
lib/api-client.ts (axios + React Query, used by useDashboard, useProgress, usePracticeSession) and lib/clientApi.ts (native fetch, used by useProfile, usePracticeMenu, and direct calls in containers) coexist. They have different auth header strategies, different error representations, and different retry logic. New developers adding features won't know which to use.
lib/api-client.ts vs lib/clientApi.ts
P2 — Architecture
CONSIST-03 — Three different loading patterns across the app
Full-screen centred spinner (practice session start), inline skeleton shimmer (dashboard recommended section), and button text swap ("…" in Ask card submit). These are used inconsistently for similar situations — all three can appear on the same screen. The app has a shared skeleton.tsx and spinner.tsx in components/ui/ that are largely ignored.
practice/PracticeView.tsx, dashboard/DashboardView.tsx, ask-card.tsx
P3 — Design System
CONSIST-04 — Button border radius mix: rounded-xl vs rounded-2xl
Sign-in/sign-up forms use rounded-xl (12px). Dashboard cards, practice buttons, and profile buttons use rounded-2xl (16px). The footer CTAs on the landing page use rounded-2xl and rounded-xl within the same nav row. Not a functional bug, but visually inconsistent enough to notice across pages.
app-wide
P3 — Visual Polish
CONSIST-05 — usePracticeSession hook is unused; PracticeContainer reimplements the same logic
hooks/use-practice-session.ts is a well-architected hook with a full state machine, React Query mutations, cache invalidation, and both hint + explanation support. PracticeContainer ignores it entirely and implements its own raw-fetch session lifecycle with fewer features (no explanation endpoint, no cache invalidation, no typed state machine). The hook and the container are two parallel implementations of the same feature.
hooks/use-practice-session.ts vs containers/PracticeContainer.tsx

Mobile UX Problems

4 issues
MOBILE-01 — XP animation overlaps content on mobile at fixed top-20
The "+N XP! ✨" float is positioned fixed top-20 right-8 (80px from top). On mobile, the page content starts at top: 0 (no header bar in practice). The float appears over the question difficulty chip and the first line of the question text, making it hard to read both simultaneously while the animation plays.
apps/web/components/mathai/practice/PracticeView.tsx : xpAnim block
P2 — Overlap
MOBILE-02 — Progress page stats row uses grid-cols-3 with no mobile override
The three stats cards (Topics mastered / Total topics / Lifetime XP) use grid grid-cols-3 gap-4 with no responsive breakpoint. On a 320px screen, each card is ~96px wide. The large font numbers (text-3xl font-black) can overflow at 4+ digit XP values. Should be grid-cols-1 sm:grid-cols-3 or use grid-cols-3 with smaller text on mobile.
apps/web/components/mathai/progress/ProgressView.tsx : stats row
P2 — Overflow Risk
MOBILE-03 — Ask page sticky input bar may be hidden by mobile keyboard
The ask page uses sticky bottom-0 with pb-safe for the textarea. When the soft keyboard appears on iOS/Android, the viewport shrinks but the sticky bar does not always scroll into the visible area correctly. The pb-safe class (for safe-area inset) helps with notched phones but doesn't account for the keyboard. This can result in the input bar being hidden behind the keyboard.
apps/web/app/ask/AskPageContent.tsx : input bar
P2 — Input Accessibility
MOBILE-04 — Practice multiple-choice uses grid-cols-2 on all screen sizes
The answer options grid is grid grid-cols-2 gap-3 with no responsive variant. Long answer text in options (e.g., "12 divided by 4 equals 3") gets truncated or wraps in awkward ways at narrow widths. On very small screens (320px), each option is ~140px — tight for 2-line option text.
apps/web/components/mathai/practice/PracticeView.tsx : multiple_choice grid
